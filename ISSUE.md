# Flyway migrations not running on Spring Boot 4.x — `flyway-core` no longer triggers auto-configuration

## Summary
Adding `org.flywaydb:flyway-core` (the standard Spring Boot 2.x/3.x setup) to a Spring Boot **4.0.6** project does not enable Flyway. The application starts normally, but no migrations run, no `flyway_schema_history` table is created, and no Flyway log lines appear at all.

## Environment
- Spring Boot: `4.0.6`
- Java: `21`
- Database: PostgreSQL `16.13`
- Flyway: `11.14.1` (managed by Spring Boot BOM)

## Symptoms
1. `flyway-core` and `flyway-database-postgresql` are present in `mvn dependency:tree`.
2. `spring.flyway.*` is configured in `application.yml`.
3. Migration files exist under `src/main/resources/db/migration/V1__init_schema.sql`.
4. Application starts successfully but:
   - No log lines containing `Flyway` / `FlywayExecutor` / `DbMigrate`.
   - No `flyway_schema_history` table is created in Postgres.
   - Even the `--debug` auto-configuration condition report (Positive matches / Negative matches) contains **zero** mentions of `FlywayAutoConfiguration`.

## Root Cause
In **Spring Boot 4.x**, Flyway auto-configuration was **moved out of `spring-boot-autoconfigure`** into its own dedicated module (`spring-boot-flyway`), exposed via the new starter **`spring-boot-starter-flyway`**.

This was confirmed by inspecting the autoconfig imports file inside the BOM-resolved jar:

```bash
unzip -p ~/.m2/repository/org/springframework/boot/spring-boot-autoconfigure/4.0.6/spring-boot-autoconfigure-4.0.6.jar \
  META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports | grep -i flyway
# (no output)
```

Because `FlywayAutoConfiguration` is no longer registered in the core auto-configuration imports file, simply having `flyway-core` on the classpath has no effect — Spring Boot never evaluates it, never creates the `Flyway` bean, and never triggers `FlywayMigrationInitializer` before the JPA `EntityManagerFactory` starts.

This is part of a broader Boot 4.x pattern where many integrations (Flyway, JDBC, JPA, security tests, etc.) are now split into per-feature `spring-boot-<feature>` + `spring-boot-starter-<feature>` modules instead of being bundled inside `spring-boot-autoconfigure`.

## Why it was easy to miss
- `mvn dependency:tree` showed Flyway present, so classpath looked correct.
- The app started without any error or warning — Flyway's absence was completely silent.
- No deprecation message or hint pointed to the new starter.
- Existing tutorials, blog posts, and StackOverflow answers (e.g. https://stackoverflow.com/questions/61619716) all assume the Boot 2.x/3.x behavior where `flyway-core` alone is sufficient.

## Solution

Replace the raw `flyway-core` dependency with `spring-boot-starter-flyway`. Keep `flyway-database-postgresql` because Flyway 10+ requires the per-database module separately.

### `pom.xml`
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-flyway</artifactId>
</dependency>
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-database-postgresql</artifactId>
</dependency>
```

### `application.yml`
```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    baseline-on-migrate: true
    baseline-version: 1
```

`baseline-on-migrate: true` + `baseline-version: 1` is required because the dev/prod databases were already populated by Hibernate `ddl-auto: update`. On first run Flyway baselines the existing schema at version 1 instead of trying to re-run `V1__init_schema.sql` (which would fail with "table already exists"). On a fresh empty database, `V1` runs normally to create the schema.

### `application-dev.yml` / `application-prod.yml`
Switch JPA from schema-managing to schema-verifying so Flyway alone owns DDL:

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
```

### `src/main/resources/db/migration/V1__init_schema.sql`
Mirrors the current Hibernate-generated schema exactly (column names follow Spring's `SpringPhysicalNamingStrategy`, which converts `userId` → `user_id`):

```sql
CREATE TABLE users (
    id       BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email    VARCHAR(255) NOT NULL UNIQUE,
    name     VARCHAR(255),
    password VARCHAR(255),
    role     VARCHAR(255) CHECK (role IN ('USER', 'ADMIN'))
);

CREATE TABLE tasks (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    created_at  TIMESTAMP(6),
    description VARCHAR(255),
    status      VARCHAR(255) CHECK (status IN ('TODO', 'IN_PROGRESS', 'DONE')),
    title       VARCHAR(255),
    user_id     BIGINT REFERENCES users(id)
);
```

And in Postgres:
```sql
SELECT installed_rank, version, description, type, success FROM flyway_schema_history;
-- 1 | 1 | << Flyway Baseline >> | BASELINE | t
```

## Takeaways
- On Spring Boot 4.x, always use `spring-boot-starter-flyway` (not `flyway-core` directly).
- Same naming pattern likely applies to other previously-bundled integrations in 4.x — when an integration silently does nothing, check whether its auto-config moved into its own `spring-boot-starter-<feature>` module.
- For migrating an existing schema into Flyway control, always pair `baseline-on-migrate: true` with an explicit `baseline-version` and a `V<baseline>__*.sql` file that reproduces the current schema for fresh databases.

## Workflow Going Forward
- Add new migrations as `V2__*.sql`, `V3__*.sql`, etc. Files run in version order on app startup.
- **Never edit a migration that has already been applied** — write a new one. Flyway records a checksum of each applied script and refuses to start if a previously-applied script changes.
